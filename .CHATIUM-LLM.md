# Проект "Чат" - Чат-приложение на основе Chatium Feed API

## Описание проекта

Полнофункциональное чат-приложение в стиле Telegram, построенное на основе модуля `@app/feed` платформы Chatium. Приложение реализовано как SPA (Single Page Application) с клиентским роутингом и real-time обновлениями через WebSocket.

## Структура и слои (006-arch)

- **README.md** — обзор, запуск, ссылки на docs.
- **config/routes.tsx** — PROJECT_ROOT, ROUTES, withProjectRoot(); обязателен для ссылок.
- **lib/** — доменная логика: logger.lib (writeServerLog, shouldLogByLevel), logger-settings.lib (уровень логирования, метрики, LOG_LEVELS, isLogLevel), messages-helpers (normalizeAuthorId, normalizeReactions, типы FeedMessageMinimal, CreatedBy).
- **repos/** — слой БД: server-logs.repo, chat-metrics.repo.
- **api/logging/** — эндпоинты уровня и счётчиков (get-level, set-level, get-metrics, reset-metrics); доступ Admin.
- **web/admin/logs** — роут админки логов (редирект на профиль → Настройки). Доступ только для роли Admin; при отсутствии прав — 403. Управление уровнем и счётчиками — в компоненте AdminSettings (Профиль → Настройки).
- **components/ui/** — переиспользуемые компоненты: AppButton, AppCard, AppSection, AppSelect, AppNotification (используются в AdminSettings и при необходимости в других экранах).
- **docs/** — architecture.md, api.md, data.md, imports.md, run.md, LLM/.

## Архитектура Feed API

### Основные сущности

#### 1. Feed (Фид/Чат)
Основная сущность чата. Содержит:
- `id` (uid) - уникальный идентификатор фида
- `title` - название чата
- `pinnedMessageId` - ID закрепленного сообщения
- `lastMessageId` - ID последнего сообщения
- `createdAt` / `updatedAt` - временные метки
- `inboxSubjectId` - уникальный субъект для inbox
- `inboxUrl` - URL для inbox
- `inboxExtraData` - дополнительные данные для inbox
- `hooks` - хуки для кастомизации поведения

#### 2. Participant (Участник)
Связь между пользователем и фидом:
- `id` - ID участника
- `userId` - ID пользователя Chatium
- `role` - роль участника (guest, admin, owner и т.д.)
- `muted` - флаг отключения уведомлений
- `inboxDisabled` - флаг отключения inbox
- `read_at` / `read_message_id` - метки прочитанных сообщений
- `data` - произвольные данные участника
- `hooks` - хуки для кастомизации

#### 3. Message (Сообщение)
Сообщение в фиде:
- `id` - ID сообщения
- `type` - тип: 'Message', 'System', 'Change', 'Blocks'
- `text` - текст сообщения
- `files` - прикрепленные файлы
- `sticker` - стикер
- `reactions` - реакции пользователей
- `reply_to` - ID сообщения, на которое отвечаем
- `as_feed` - флаг отправки от имени фида
- `data` - произвольные данные
- `createdBy` / `createdAt` / `updatedAt` - метаданные

### Основные операции API

#### Управление фидами
- `createFeed(ctx, props)` - создать новый фид
- `getFeedById(ctx, id)` - получить фид по ID
- `updateFeed(ctx, props)` - обновить фид
- `deleteFeed(ctx, feedOrUid)` - удалить фид
- `getChat(ctx, feed)` - получить ChatProps для отображения
- `setFeedPinnedMessage(ctx, feed, messageId)` - закрепить сообщение

#### Управление участниками
- `getOrCreateParticipant(ctx, feed, user, options)` - получить или создать участника
- `createOrUpdateFeedParticipant(ctx, feed, user, options)` - создать/обновить участника
- `deleteFeedParticipant(ctx, feed, participant)` - удалить участника
- `findFeedParticipants(ctx, feed, options)` - найти участников
- `getFeedParticipantsCount(ctx, feed)` - получить количество участников

#### Управление сообщениями
- `createFeedMessage(ctx, feed, author, message, options)` - создать сообщение
- `findFeedMessages(ctx, feed, options)` - найти сообщения (head/tail/around)
- `findFeedMessageById(ctx, feed, id)` - найти сообщение по ID
- `updateFeedMessage(ctx, feed, messageId, data)` - обновить сообщение
- `deleteFeedMessage(ctx, feed, messageId)` - удалить сообщение
- `setFeedPinnedMessage(ctx, feed, messageId)` - закрепить сообщение

### Логические связи

```
User (SmartUser)
  └── Participant (связь многие-ко-многим через Feed)
        ├── role
        ├── muted
        └── read_message_id

Feed
  ├── Participants (1-ко-многим)
  ├── Messages (1-ко-многим)
  ├── pinnedMessage
  └── lastMessage

Message
  ├── createdBy → User
  ├── reply_to → Message
  ├── reactions → {emoji: User[]}
  └── files[]
```

### AI-Агенты в чатах

Система интеграции с AI-агентами Chatium через `pushMessageToChain`. Позволяет добавлять системных агентов в групповые чаты.

#### Архитектура

**Как это работает:**
1. Администратор выбирает системного агента из списка агентов аккаунта (получается через `findAgents`)
2. Для агента создается бот-пользователь в Chatium (через `createOrUpdateBotUser`)
3. Бот-пользователь добавляется как участник чата
4. При новых сообщениях в чате система проверяет критерии ответа
5. Если критерии совпадают — сообщение отправляется агенту через `pushMessageToChain`
6. Агент обрабатывает сообщение в своей цепочке и может использовать тулы для ответа

#### Таблица chat_agents

Хранит связь между системными агентами и чатами:
- `agentId` — ID агента в системе Chatium
- `agentKey` — ключ/слаг агента
- `agentName` — отображаемое имя
- `botUserId` — ID созданного бот-пользователя
- `chainKey` — ключ цепочки сообщений для этого чата
- `respondTo` — условие ответа ('all' | 'admins' | 'mention')
- `respondToMention` — кто может упоминать ('all' | 'admins')

#### Режимы ответов агента

| Режим | Описание |
|-------|----------|
| `all` | Агент отвечает на все сообщения в чате |
| `admins` | Агент отвечает только на сообщения от администраторов |
| `mention` | Агент отвечает только при упоминании по имени (@Имя) |

#### Тулы для агентов

**`replyInGroupChat`** — отправка текстового сообщения:
```typescript
await replyInGroupChat.run(ctx, {
  context: {},
  input: {
    chatId: 'feed-id',
    text: 'Текст ответа',
    replyToMessageId: 'optional-message-id'
  }
});
```

**`sendImageToGroupChat`** — отправка изображения:
```typescript
await sendImageToGroupChat.run(ctx, {
  context: {},
  input: {
    chatId: 'feed-id',
    imageUrl: 'https://fs.chatium.ru/get/xxx',
    caption: 'Подпись',
    replyToMessageId: 'optional-message-id'
  }
});
```

⚠️ **ВАЖНО**: Агент должен использовать ТОЛЬКО ОДИН инструмент за раз — либо текст, либо изображение!

#### Обработка сообщений

При отправке сообщения в групповой чат:
1. Система загружает активных агентов из `chat_agents`
2. Проверяет `respondTo` и `respondToMention` для каждого агента
3. Формирует контекстное сообщение с историей чата
4. Отправляет агенту через `pushMessageToChain` с `wakeAgent: true`
5. Агент получает полный контекст: имя отправителя, роль, текст, история сообщений

### SPA Архитектура

Приложение реализовано как Single Page Application с клиентским роутингом:

#### Корневой компонент: `App.vue`
- Управляет текущим view (chats / profile / settings)
- Содержит глобальную модалку создания чата
- Синхронизирует состояние с URL hash (`#/chat/:feedId`)
- Подписывается на WebSocket события пользователя
- Обновляет список чатов при новых сообщениях

#### Composables:

**`useChatSocket.ts`** - Composable для работы с WebSocket
- Подписка на события пользователя
- Автоматическое переподключение
- Обработка событий чата и приглашений

**`ChatsList.vue`** - Список чатов (сайдбар)

- **Кастомные папки** — создание, редактирование, удаление папок
- **Drag & Drop** — упорядочивание фильтров перетаскиванием
- Список приглашений (InvitesList)
- Меню профиля

- Превью последнего сообщения с именем автора

**`ChatView.vue`** - Область чата
- Шапка с информацией о чате и typing indicator
- Закрепленное сообщение (PinnedMessage)
- История сообщений с real-time обновлениями
- Реакции на сообщения (MessageReactions)
- Ответы на сообщения (MessageReply)

## Личные чаты (Direct Chats)

Личные чаты — это чаты типа `direct`, предназначенные для общения один-на-один между двумя пользователями.

### Особенности личных чатов:
- **Только два участника** — в личном чате всегда ровно два участника (вы и собеседник)
- **Нет приглашений** — нельзя пригласить третьего участника, личный чат строго на двоих
- **Нет списка участников** — в профиле чата не показывается список участников и кнопка "Пригласить"
- **Автоматическое имя** — название чата = имя собеседника (из его профиля)
- **Аватар собеседника** — в качестве аватарки чата используется аватарка собеседника
- **Нет редактирования** — нельзя изменить название, описание или аватарку чата (они берутся из профиля собеседника)
- **Нет публичности** — личный чат не может быть публичным
- **Нет платности** — личный чат не может быть платным
- **Блокировка** — доступна кнопка "Заблокировать пользователя" вместо управления участниками

### Интерфейс личного чата:
- В шапке чата скрыты кнопки "Пригласить" и "Участники"
- В модалке информации о чате скрыты:
  - Метка "Личный чат"
  - Список участников
  - Кнопка "Пригласить участника"
  - Кнопка "Редактировать чат"
  - Кнопка "Сделать платным"
- В мобильном меню скрыты соответствующие пункты
- Панель участников (`ParticipantsPanel`) не отображается для личных чатов

Приложение использует WebSocket для мгновенной доставки событий:

### Архитектура WebSocket

#### На сервере (API):
Используется модуль `@app/socket` для отправки событий:

```typescript

// Генерация encoded socket ID для пользователя
const encodedSocketId = await genSocketId(ctx, `user-${userId}`)
```

## Особенности реализации

### Авторизация
Используется встроенная система авторизации Chatium:
- `ctx.user` - текущий пользователь
- `requireRealUser(ctx)` - требование авторизации
- Редирект на `/s/auth/signin` для неавторизованных

### Безопасность и система прав

#### Роли участников
В чате существует три уровня доступа:

| Роль | Права |
|------|-------|
| **owner** (Владелец) | Полный контроль: удаление чата, изменение настроек публичности, назначение админов, изменение ролей |
| **admin** (Администратор) | Закрепление сообщений, добавление/удаление участников (кроме owner), редактирование чата |
| **guest** (Участник) | Отправка сообщений, реакции, ответы на сообщения |

#### Управление ролями
Только владелец чата может изменять роли участников через панель участников:
- Повышение до администратора (`guest` → `admin`)
- Понижение до участника (`admin` → `guest`)
- Выбор роли при добавлении нового участника

#### Проверка прав
Используется вспомогательная функция `canManageChat()` из `shared/permissions.ts`:
```typescript
import { canManageChat, isChatOwner } from '../shared/permissions'

// Проверка на админа/владельца
const canManage = await canManageChat(ctx, feedId, userId)

// Проверка на владельца
const isOwner = await isChatOwner(ctx, feedId, userId)
```

#### Особенности:
- Владелец не может быть удалён из чата
- Только владелец может назначать администраторов и изменять роли
- Только владелец может менять настройку публичности чата
- Сам себя может удалить любой участник
- Права проверяются по `participant.role` из Feed API
- Нельзя изменить роль самому себе или другому владельцу

### Каналы (Channels)
Каналы — это особый тип чата (`type: 'channel'`), предназначенный для one-to-many коммуникации:

#### Особенности каналов:
- **Только публикация**: Сообщения могут публиковать только владелец и администраторы
- **Подписчики**: Участники называются "подписчиками", действие присоединения — "подписаться"
- **Скрытый список**: Подписчики не видят полный список участников (только owner/admin)
- **Терминология**: В интерфейсе используются термины "подписчики" / "подписаться" вместо "участники" / "присоединиться"

#### Проверка прав в канале:
```typescript
// API: Проверка перед отправкой сообщения
if (chat.type === 'channel') {
  const role = participant?.role || 'guest'
  if (role !== 'owner' && role !== 'admin') {
    throw new Error('В канале публиковать сообщения могут только владелец и администраторы')
  }
}

// Client: Скрытие поля ввода для подписчиков
const canPostMessages = computed(() => {
  if (!isChannel.value) return true
  return isOwner.value || isAdmin.value
})
```

#### API для каналов:
- `POST /api/messages/:feedId/send` — проверяет роль в канале
- `GET /api/participants/:feedId/list` — для подписчиков возвращает только owner/admin

### Выход из чата и отписка от канала

#### Выход из группового чата:
- Доступно любому участнику кроме владельца
- Удаляет пользователя из списка участников
- Чат исчезает из списка чатов пользователя

#### Отписка от канала:
- Доступна любому подписчику кроме владельца
- Аналогично выходу из чата, но с терминологией каналов
- Владелец не может отписаться — только удалить канал

#### API:
```typescript
// Выход из группового чата
await apiChatLeaveRoute.run(ctx, { feedId });

// Отписка от канала
await apiChannelUnsubscribeRoute.run(ctx, { feedId });
```

### Чёрный список (блокировка пользователей)

Система блокировки для личных чатов:

#### Возможности:
- **Блокировка пользователя** — запрет на получение сообщений от заблокированного пользователя
- **Скрытие чатов** — личные чаты с заблокированными пользователями скрываются из списка
- **Запрет отправки** — нельзя отправить сообщение, если один из участников заблокировал другого
- **Управление списком** — просмотр и разблокировка пользователей через профиль

#### Механизм работы:
```
Пользователь А блокирует Пользователя Б:
→ Чат между ними исчезает из списка чатов А
→ Б не может писать новые сообщения А
→ А не может писать новые сообщения Б (двусторонняя блокировка)
→ При попытке отправки — ошибка "Вы заблокировали этого пользователя"
```

#### Интерфейс:
- В профиле добавлена вкладка «Чёрный список» с возможностью разблокировки
- В личных чатах доступна кнопка «Заблокировать пользователя» в информации о чате

#### API:
```typescript
// Получить список заблокированных
const { blockedUsers } = await apiBlockedUsersListRoute.run(ctx);

// Заблокировать пользователя
await apiBlockedUsersBlockRoute.run(ctx, { 
  userId: 'user-id',
  reason: 'Причина блокировки' // опционально
});

// Разблокировать пользователя
await apiBlockedUsersUnblockRoute.run(ctx, { userId: 'user-id' });

// Проверить статус блокировки
const { isBlocked } = await apiBlockedUsersCheckRoute.run(ctx, { userId: 'user-id' });
```

### Публичные чаты
Публичные чаты (`isPublic: true`) доступны всем пользователям:
- **В списке чатов** — публичные чаты видны всем пользователям
- **Поиск** — публичные чаты включаются в результаты поиска
- **Просмотр** — не-участники видят информацию о чате, но не сообщения
- **Присоединение** — кнопка "Войти" или "Присоединиться" для вступления в чат
- **Автоматическое добавление** — при присоединении пользователь становится участником с ролью `guest`

#### Ограничение на создание публичных чатов
Только **администраторы воркспейса** (роль `Admin` в Chatium) могут:
- Создавать публичные чаты (устанавливать `isPublic: true` при создании)
- Делать существующий чат публичным (изменять `isPublic` на `true`)

Обычные пользователи (`Staff`, `User`) могут создавать только приватные чаты.

#### API для публичных чатов:
- `POST /api/chats/create` — требует `Admin` роль для `isPublic: true`
- `GET /api/chats/list` — возвращает публичные чаты даже если пользователь не участник
- `GET /api/chats/:feedId` — доступ к информации о публичном чате
- `POST /api/chats/:feedId/update` — требует `Admin` роль для изменения `isPublic` на `true`
- `POST /api/chats/:feedId/join` — присоединиться к публичному чату
- `GET /api/messages/:feedId/list` — возвращает пустой список для не-участников публичного чата

### Система приглашений
Поддерживается 4 способа приглашения в чат:

1. **По username** - поиск пользователя по логину (@username)
2. **По email** - поиск по подтвержденному email адресу
3. **По телефону** - поиск по подтвержденному номеру телефона
4. **Инвайт-ссылка** - универсальная ссылка для любого пользователя

#### Жизненный цикл приглашения:
```
Создание → Отправка → Принятие/Отклонение
    ↓
Приглашение по ссылке: срок действия 7 дней
Приглашение конкретному пользователю: бессрочно
```

#### Нормализация реакций
Поскольку `reactions` хранятся в JSON поле базы данных, при загрузке из Feed API они могут приходить как строка вместо объекта. Для защиты от этого реализована многоуровневая нормализация:

**На сервере (`api/messages.ts`, `api/reactions.ts`):**
```typescript
function normalizeReactions(message: any): any {
  let reactions = message.reactions || message.data?.reactions || {}
  if (typeof reactions === 'string') {
    try {
      reactions = JSON.parse(reactions)
    } catch (e) {
      reactions = {}
    }
  }
  return reactions
}
```

**На клиенте (`ChatView.vue`, `MessageReactions.vue`):**
- `normalizeMessageFields()` — нормализует реакции при загрузке сообщений
- `getMessageReactions()` — безопасное получение реакций
- `safeReactions` (computed) — нормализованные реакции для шаблона
- `onReactionUpdate()` / `handleReactionUpdate()` — защита при обновлении через WebSocket

### Счетчики непрочитанных сообщений

Реализована система отображения непрочитанных сообщений с использованием `@app/inbox`:

#### Архитектура:
- **Inbox API** — централизованная система для хранения badge-счетчиков
- **Badge отображение** — зеленый кружок с числом непрочитанных рядом с названием чата
- **Автоматический сброс** — при открытии чата badge обнуляется
- **Real-time обновление** — WebSocket события обновляют счетчики

#### API:
```typescript
// Получить все badges пользователя
const { badges } = await apiInboxBadgesGetRoute.run(ctx)
// badges: { [subjectId]: number }

// Сбросить badge для чата
await apiInboxBadgeResetRoute.run(ctx, {
  subjectId: feed.inboxSubjectId,
  url: feed.inboxUrl,
})
```

#### Поток данных:
```
Новое сообщение → Feed API обновляет inbox → WebSocket уведомление
                                                    ↓
                                          App.vue загружает badges
                                                    ↓
                                          ChatsList отображает badge
```

#### Сброс badge:
- **При открытии чата** — автоматический сброс в `ChatView.loadChat()`
- **При прокрутке вниз** — badge также сбрасывается
- **Событие родителю** — `@badge-reset` для обновления UI

#### Интеграция с Feed API:
- Каждый feed имеет `inboxSubjectId` и `inboxUrl`
- При создании фида эти поля устанавливаются автоматически
- Feed API автоматически обновляет inbox при новых сообщениях

#### Компоненты:
- **App.vue** — загружает inbox badges через `loadInboxBadges()`
- **ChatsList.vue** — отображает badge через `getBadgeCount(chat)`
- **ChatView.vue** — сбрасывает badge через `resetChatBadge()`
- **api/inbox-badges.ts** — API routes для работы с badges

| Оформление подписки | Любой авторизованный пользователь |
| Отмена подписки | Только владелец подписки |

**Важно:** Платежи работают на уровне воркспейса, поэтому только администраторы воркспейса (роль `Admin`) могут управлять платным функционалом. Владельцы чатов без роли Admin не могут создавать платные чаты или управлять тарифами.

#### API:
- `GET /api/chat-subscription-plans/:feedId/plans` — список активных тарифов (для всех)
- `GET /api/chat-subscription-plans/:feedId/plans/all` — все тарифы включая неактивные (только Admin)
- `POST /api/chat-subscription-plans/:feedId/plans/create` — создать тариф (только Admin)
- `POST /api/chat-subscription-plans/:feedId/plans/:id/update` — обновить тариф (только Admin)
- `POST /api/chat-subscription-plans/:feedId/plans/:id/delete` — удалить тариф (только Admin)
- `POST /api/chat-subscriptions/:feedId/subscribe` — оформить подписку
- `POST /api/chat-subscriptions/:feedId/cancel` — отменить автопродление
- `POST /api/chat-subscriptions/:feedId/check-access` — проверить доступ к чату

#### Особенности календарных периодов:
- При покупке подписки на месяц пользователь видит выбор текущего/следующего месяца
- Для кварталов (например, май-июль) — выбор конкретного периода
- Доступ строго в рамках выбранного периода

#### Автопродление:
- Пользователь явно соглашается на автопродление (checkbox)
- В профиле отображается дата следующего списания
- Возможность отмены в любой момент (доступ сохраняется до окончания периода)

## Динамические пути и переносимость приложения

Приложение спроектировано для работы в любом окружении Chatium независимо от пути развёртывания.

### Архитектура динамических путей:

#### Бэкенд (`shared/app-paths.ts`):
- `getAppBasePath(ctx)` — определяет базовый путь из URL запроса
- `getAppUrl(ctx, path)` — генерирует URL относительно базового пути
- `getFullUrl(ctx, path)` — генерирует полный URL с origin

#### Фронтенд (`composables/useApiUrl.ts`):
- `useApiUrl()` — composable для генерации API URL
- `apiUrl(endpoint)` — хелпер для генерации относительных API путей
- Использует относительные пути (`./api/...`) для корректной работы в любом окружении

### Исправленные хардкод-пути:

| Файл | Было | Стало |
|------|------|-------|
| `api/invites.ts` | `/projekt-chat/invite~${token}` | `getFullUrl(ctx, /invite~${token})` |
| `index.tsx` | `/projekt-chat/manifest.json` | `getAppUrl(ctx, /manifest.json)` |
| `index.tsx` | `/projekt-chat/styles/*.css` | `getAppUrl(ctx, /styles/*.css)` |
| `invite.tsx` | `/projekt-chat/invite~${token}` | `invitePath` (динамический) |
| `manifest.json.ts` | `/projekt-chat/` | `getAppUrl(ctx, /)` |
| Vue компоненты | `/projekt-chat/api/*` | `makeApiUrl(endpoint)` |

### Рекомендации по добавлению новых маршрутов:

1. **На бэкенде**: Используйте `getAppUrl()` и `getFullUrl()` из `shared/app-paths.ts`
2. **Во Vue компонентах**: Используйте `useApiUrl()` composable
3. **Избегайте**: Абсолютных путей с `/projekt-chat/` или `/project-chat/`
4. **Для редиректов**: Используйте относительные пути или `indexPageRoute.url()`
